OBJ-5: Complete MOCKMATE-AI Application Flow
Implement the end-to-end job application flow in mockmate-ai, covering the entire journey from the Apply button click through fitness scoring, Zinterview backend validation, candidate registration, and scheduling-mode-aware success states.
Proposed Changes
Backend: Fitness Score Service
[MODIFY] 
fitnessScoreService.js
Complete rewrite of the fitness scoring engine:
Model upgrade: Switch from gemini-1.5-flash → gemini-2.5-flash
5-parameter weighted scoring matching the prompt spec:
Skills Match (35%), Experience Relevance (20%), Education Fit (10%), Role Alignment (20%), Keyword Density (15%)
Structured output: Return score 0–100, per-parameter breakdown, and recommendation label (STRONG_MATCH | GOOD_MATCH | FAIR_MATCH | LOW_MATCH)
MongoDB-based caching: Hash-based cache using 
(resumeId + openingId) as key with 7-day TTL
GPTCache is Python-only, so we use a practical JS equivalent: hash the 
(resumeText + jobDescription) pair, store in a FitnessScoreCache collection with TTL index
For "semantic" similarity: normalize the resume+JD text, produce a deterministic hash, and check for exact match first; for near-matches, store a text fingerprint (top-N TF-IDF terms sorted) and compare fingerprints
Persistent score storage: After calculation, save the score directly on the JobApplication document with full breakdown
Cost optimization: Truncate resume text to 4000 chars and JD to 2500 chars to minimize tokens while retaining accuracy

Backend: Application Controller
[MODIFY] 
jobApplicationController.js
Enhance the submit flow with the full OBJ-5 pipeline:
POST /api/applications/submit-full — New endpoint that:
Checks if opening is still active
Checks for existing application (dedup)
Extracts resume text from S3 (via 
CandidateResume lookup)
Calls fitnessScoreService.calculateFitnessScore()
Returns fitness result to frontend for display
POST /api/applications/finalize — After user reviews score:
Validates fitness score against opening's mockmateConfig.fitnessThreshold
If failed → returns rejection with appropriate message
If passed → creates the JobApplication record with full fitness breakdown
Calls ZinterviewService.createCandidate() to register on Zinterview
Stores the returned interviewReportId on the application
Returns scheduling mode + interview URL for frontend handling
Save profile logic: On savedDetails: true, upsert 
CandidateProfile with all UserDetails.jsx-matching fields
Saved-details fast-path: If CandidateProfile.savedDetails === true and all fields populated, allow direct apply (skip form, show confirmation toast)
